## Stage 1.A — Schema 基础设施

**改动**：
- `src/cloud/sqlite-store.js` 在 schema 段加：
  ```sql
  CREATE TABLE IF NOT EXISTS user_events (
    id          INTEGER PRIMARY KEY,
    user_id     TEXT NOT NULL,
    seq         INTEGER NOT NULL,
    kind        TEXT NOT NULL,
    scope_kind  TEXT,
    scope_ref   TEXT,
    payload     TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    UNIQUE (user_id, seq)
  );
  CREATE INDEX IF NOT EXISTS idx_user_events_user_seq ON user_events(user_id, seq);

  CREATE TABLE IF NOT EXISTS op_idempotency (
    user_id     TEXT NOT NULL,
    client_op   TEXT NOT NULL,
    result_json TEXT NOT NULL,
    created_at  TEXT NOT NULL,
    PRIMARY KEY (user_id, client_op)
  );
  ```
- `users` 表 ALTER TABLE 加 `event_seq INTEGER NOT NULL DEFAULT 0`
- Migration row 记录在 schema_migrations
- 新文件 `src/cloud/event-log-store.js` 暴露 `appendEvent(userId, kind, scopeKind, scopeRef, payload)` → 返回写入的 event。**事务内**完成 INSERT + UPDATE users.event_seq

## Stage 1.B — broadcastEvent 改为持久化 + 串联 seq

**改动 `scripts/serve-cloud.js`**:
- `broadcastEvent(hub, userId, payload)` 改为：先 `eventLog.appendEvent(userId, payload.type, ...)` 同事务拿 seq，再 `payload.seq = seq`，再 socket push
- 哪些事件该持久化（**state changes**）：
  - `social.friend_request_received`
  - `social.friend_added`
  - `social.room_invited`
  - `room.message_appended`
  - `room.updated`
  - `room.deleted`
  - `room.fellow_invocation_requested`
  - `workspace_updated`
  - `message_created`
- 哪些**不需要**持久化（transient/runtime）：
  - `bridge_run_updated` / `bridge_ready` / `bridge_run_event`（agent 任务流，纯实时）
  - `device_updated`（设备 online/offline）
  - `events_ready`（握手）
- 提供 `broadcastEventTransient(hub, userId, payload)` 给不需持久化的事件用，避免误用

**验收**：
```bash
# 跑所有 cloud / serve / room 测试，无 regression
node --test tests/cloud-*.test.js tests/serve-cloud-*.test.js tests/main-social-api.test.js | tail -5

# 测试：发一条 room 消息 → user_events 出现一行 + seq = users.event_seq
node --test tests/event-log-persisted.test.js
```

## Stage 1.C — WS 握手支持 `since_seq`，自动回放

**改动 `scripts/serve-cloud.js`**:
- WS upgrade handler 已读 `?token=`，加 `?since_seq=N` 解析
- 连接 attach 后立即查 `user_events WHERE user_id=? AND seq > since_seq ORDER BY seq LIMIT 500`，flush 给该 socket
- 如果 user_events 数量 > 500，分批 flush；最后一条带 `more: true` 标记
- 客户端可主动 GET `/api/me/events?since_seq=N&limit=500` 拉更早的（fallback）

**改动 `src/main.js` (cloud client)**:
- WS 连接 URL 拼接 `since_seq=last_seen_seq`（默认 0）
- 收事件后存 `seq` → settings 持久化 `lastEventSeq`
- 重连读 last seq → 拼上去

**改动 `src/web/app.js`**:
- 同上：localStorage `aimashi.lastEventSeq.<account>`
- WS URL 带 since_seq

**验收**：
```bash
# 集成测试：模拟断连 → 服务器 5 条 event → 重连 → 客户端拿到 5 条
node --test tests/sync-replay.test.js

# Manual：开两个客户端，A 关闭后 B 在 A 关联的 room 发 5 条消息，A 启动后必须立刻看到 5 条
```

## Stage 1.D — 写操作幂等 via `clientOpId`

**改动 `scripts/serve-cloud.js`**:
- 新增 middleware 在所有写 endpoint（POST/PATCH/DELETE）前生效
- 如 body 含 `clientOpId`：
  - SELECT op_idempotency WHERE user_id=? AND client_op=? → 命中直接返回 result_json
  - 否则正常处理 + 处理完 INSERT op_idempotency (user_id, client_op, JSON result)
- 老化清理：每小时定时清 created_at < now-24h

**改动客户端**：
- `src/main/social/social-api.js`：所有 POST/PATCH/DELETE 调用前生成 uuid 作 clientOpId 附 body
- `src/web/app.js` 同
- 服务器返回的成功响应客户端记入临时 cache (5 min TTL)，重发时复用 ID

**验收**：
```bash
# 测试：同 clientOpId 重复 POST /api/rooms → 第二次返回第一次的 room
node --test tests/op-idempotency.test.js
```

## Stage 1.E — 客户端缓存 last_seen_seq

**改动**：
